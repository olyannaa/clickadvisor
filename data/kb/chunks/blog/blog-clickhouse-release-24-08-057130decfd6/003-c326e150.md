WeвҖҷll use the materialized column as the sorting key. The table creation query is shown below: ``` CREATE TABLE events ( matchId String, json JSON, possession_team_id String MATERIALIZED getSubcolumn(json, 'possession_team.id') ) ENGINE = MergeTree ORDER BY possession_team_id; ```

We can then copy our previous `SELECT` query and prefix it with `INSERT INTO events` to load the data:

```
INSERT INTO events 
SELECT
    replaceRegexpAll(splitByRegexp('/', _file)[-1], '.json', '') AS matchId,
    json
FROM file('master.zip :: **/data/events/*.json', JSONAsObject)
```

It takes a little over one minute from the output to load 12 million events. We can then write a query that uses JSON dot syntax to find the most popular types of events:

```
SELECT
    json.type.name,
    count() AS count
FROM events
GROUP BY ALL
ORDER BY count DESC
LIMIT 10
```

The dot syntax is useful when we want to read literal values, but it doesnвҖҷt work if we read subobjects. For example, the following query counts the most popular `possesion_team`:

```
SELECT
    json.possession_team AS team,
    count()
FROM events
GROUP BY team
ORDER BY count() DESC
LIMIT 10
```

Hmmm, all nulls!

The dot syntax doesnвҖҷt read nested objects for performance reasons. The data is stored so that reading literal values by paths is very efficient, but reading all subobjects by path requires much more data and can sometimes be slower.

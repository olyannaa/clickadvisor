### Hardware and system versions [\#](/blog/elasticsearch-log-analytics-clickhouse#hardware-and-system-versions) Both systems were benchmarked on **identical, isolated single\-node environments** to ensure a fair apples\-to\-apples comparison. Each engine ran on its own dedicated EC2 instance with the same CPU, memory, and storage configuration.

| System | Instance | CPU / RAM | Storage | Version |
| --- | --- | --- | --- | --- |
| ClickHouse | AWS EC2 m6i.8xlarge | 32 vCPUs / 128 GiB RAM | gp3 (16k IOPS, 1,000 MB/s) | OSS v26\.3 |
| Elasticsearch | AWS EC2 m6i.8xlarge | 32 vCPUs / 128 GiB RAM | gp3 (16k IOPS, 1,000 MB/s) | OSS v9\.3\.2 |


### System configuration [\#](/blog/elasticsearch-log-analytics-clickhouse#system-configuration)

#### ClickHouse [\#](/blog/elasticsearch-log-analytics-clickhouse#clickhouse)

ClickHouse used a standard MergeTree table with the OpenTelemetry log [schema](https://github.com/ClickHouse/TextBench/blob/main/clickhouse/create.sql), including native data types for timestamps, numerics, strings, and maps. The table was [ordered](https://github.com/ClickHouse/TextBench/blob/a7bb4e024f4648e90a30b6f0d00ef1223fd7d7e5/clickhouse/create.sql#L22) by:

```

```
(ServiceName, Timestamp)
```

```

This sorting key improves locality for common log\-analysis queries that filter by service and time range, while also improving compression.

A [full\-text index](https://clickhouse.com/blog/clickhouse-full-text-search-object-storage) was [created](https://github.com/ClickHouse/TextBench/blob/a7bb4e024f4648e90a30b6f0d00ef1223fd7d7e5/clickhouse/create.sql#L19) on the `Body` column.

#### Elasticsearch [\#](/blog/elasticsearch-log-analytics-clickhouse#elasticsearch)

Elasticsearch used an equivalent [index mapping](https://github.com/ClickHouse/TextBench/blob/a7bb4e024f4648e90a30b6f0d00ef1223fd7d7e5/elasticsearch/create_indexes.sh#L44) with the same logical schema and matching [physical sort order](https://www.elastic.co/docs/reference/elasticsearch/index-settings/sorting):

```

```
index.sort = (ServiceName ASC, Timestamp ASC)
```

```

Key mapping choices:

- `Body` mapped as `text` with the standard analyzer (full\-text searchable)
- Structured string dimensions mapped as `keyword`
- Map\-like attribute fields mapped as `flattened`
- `TraceFlags` and `SeverityNumber` mapped as `byte`
- `codec: best_compression` enabled to minimize on\-disk footprint

This mirrors ClickHouse semantics as closely as Elasticsearch allows while following storage best practices.

To ensure stable single\-node performance, we used these [JVM](https://github.com/ClickHouse/TextBench/blob/main/elasticsearch/config/jvm.options) and OS settings

- Heap fixed at `30 GiB` (`-Xms30g -Xmx30g`) to retain CompressedOops
- `bootstrap.memory_lock: true`
- `vm.max_map_count=262144`

### Shard layout [\#](/blog/elasticsearch-log-analytics-clickhouse#shard-layout)

The benchmark used **single\-node storage\-optimized layouts** with no replicas in either system.

Per dataset, ClickHouse used a single shard (single table), and Elasticsearch used multiple shards, following [best practices for optimal shard size](https://www.elastic.co/docs/deploy-manage/production-guidance/optimize-performance/size-shards#shard-size-recommendation) (50GB per shard).

All Elasticsearch shards were force\-merged into a single segment per shard before measurement (following [best practices](https://www.elastic.co/docs/reference/elasticsearch/index-lifecycle-actions/ilm-forcemerge)). This improves storage efficiency and does not prevent parallel search execution, because [Lucene can still partition a single segment logically at query time](https://www.elastic.co/search-labs/blog/apache-lucene-10-release-highlights).

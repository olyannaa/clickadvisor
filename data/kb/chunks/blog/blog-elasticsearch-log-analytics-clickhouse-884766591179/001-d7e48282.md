# Do you still need Elasticsearch for log analytics? ClickHouse says no.

> **TL;DR**  
> ClickHouse now has a new full\-text search built for analytics and observability, combining fast multi\-token search with large\-scale aggregation in a single engine.  
>   
> That makes it a compelling alternative to Elasticsearch for log analytics.  
> This benchmark shows why.

## The last remaining moat closes [\#](/blog/elasticsearch-log-analytics-clickhouse#the-last-remaining-moat-closes)

For years, the choice was simple: Elasticsearch for text search, ClickHouse for analytics.

Now ClickHouse does both.

ClickHouse now [includes](https://clickhouse.com/blog/full-text-search-ga-release) a [redesigned](https://clickhouse.com/blog/clickhouse-full-text-search-object-storage) full\-text search powered by inverted indexes. But unlike traditional search engines, matching documents are fed directly into the same vectorized analytical engine that already [excels](https://clickhouse.com/blog/clickhouse_vs_elasticsearch_the_billion_row_matchup) at filtering, aggregations, and large\-scale scans.

That combination matters most when search is only the first step. In [observability](https://clickhouse.com/clickstack), especially log analytics, search is rarely just about finding text. Users search for an error pattern, then count it's occurrences, group it by service, drill into a more narrow time frame, or isolate affected hosts.

In other words: **logs are analytical data that happen to contain text**.

In this benchmark, we compare the OSS versions of ClickHouse and Elasticsearch on a realistic OpenTelemetry log workload across datasets up to 50 billion rows.

> **Teaser**: Across all tested dataset scales, ClickHouse runs the full\-text analytical workload **2\-6x faster than Elasticsearch**. ClickHouse also stores the OTel log dataset far more compactly than Elasticsearch.

The full benchmark is reproducible end to end: source data, schemas, queries, ingestion scripts, and measurements are published in our [public GitHub repository](https://github.com/ClickHouse/TextBench/tree/main).

We now describe the benchmark setup and methodology, then examine how both systems store OTel logs on disk before comparing storage footprint and query runtimes.

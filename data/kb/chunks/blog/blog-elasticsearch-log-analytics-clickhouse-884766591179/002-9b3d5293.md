### Dataset [\#](/blog/elasticsearch-log-analytics-clickhouse#dataset)

The benchmark uses a realistic [OpenTelemetry](https://opentelemetry.io/) log dataset generated by our [OpenTelemetry demo application](https://play-clickstack.clickhouse.com/search), producing structured service logs similar to those found in modern observability pipelines.

To make the benchmark fully reproducible, the source data is published as **Parquet files in a public Amazon S3 bucket**.

- **Region:** eu\-west\-3 (Europe — Paris)
- **Public bucket:** [s3://public\-pme/text\_bench](javascript:void(0))
- **Direct URL:** [https://public\-pme.s3\.eu\-west\-3\.amazonaws.com/text\_bench/part\_{000\..009}.parquet](https://public-pme.s3.eu-west-3.amazonaws.com/text_bench/part_%7B000..009%7D.parquet)
- **File layout:** each Parquet file contains **1 billion log rows**, sorted by timestamp
- **Tip:** run EC2 benchmark instances in **eu\-west\-3** to avoid cross\-region ingest latency and egress charges

From this source data, we loaded identical benchmark datasets into both ClickHouse and Elasticsearch at three scales:

| Data set | Rows |
| --- | --- |
| Small | 1 billion |
| Medium | 10 billion |
| Large | 50 billion |


This lets us evaluate how both systems behave from single\-billion\-row deployments up to **large\-scale production observability workloads**.

### Queries [\#](/blog/elasticsearch-log-analytics-clickhouse#queries)

To evaluate realistic full\-text analytical search performance, we use a suite of **nine OpenTelemetry log queries**, each built around single\- or multi\-token full\-text search and often followed by filtering, counting, grouping, ranking, or time\-based aggregation.

The queries are implemented for both systems using their native query languages (**SQL** for ClickHouse; **Query DSL** and **ES\|QL** for Elasticsearch).

The suite is designed to reflect **typical log\-analysis workflows** in tools like [Kibana](https://www.elastic.co/kibana) or [ClickStack](https://clickhouse.com/clickstack), where users search for an error message, then immediately filter, count, group, rank, or analyze it over time.

### Inverted indexes for full\-text search [\#](/blog/elasticsearch-log-analytics-clickhouse#inverted-indexes-for-full-text-search)

Both systems use an **② [inverted index](https://en.wikipedia.org/wiki/Inverted_index)** as the basic data structure for full\-text search.

### Stored fields and \_source [\#](/blog/elasticsearch-log-analytics-clickhouse#stored-fields-and-_source)

In Elasticsearch **③ [stored fields](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/mapping-store)**  serve as a document store for returning original field values in query responses. By default, they also store [\_source](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/mapping-source-field), which contains the original ingested JSON documents. Stored fields are compressed using the algorithm defined by the [index.codec](https://www.elastic.co/docs/reference/elasticsearch/index-settings/index-modules) setting: lz4 by default, or zstd for higher compression ratios at the cost of slower performance.

> **Why \_source matters for logs**  
> For log analytics, returning the original ingested document is often essential. In Elasticsearch OSS, that makes \_source hard to avoid in practice; Enterprise tiers can instead use [synthetic \_source](https://www.elastic.co/guide/en/elasticsearch/reference/current/mapping-source-field.html#synthetic-source).

In ClickHouse, there is no separate equivalent structure: original rows are reconstructed directly from the values stored in the column files.

### Sparse indexing for pruning [\#](/blog/elasticsearch-log-analytics-clickhouse#sparse-indexing-for-pruning)

ClickHouse uses sparse indexing: because OTel log records are stored on disk in sorted order, the engine can group them into blocks, record value ranges for each block in a **③ [sparse primary index](https://clickhouse.com/docs/primary-indexes)**, and skip blocks that fall entirely outside the requested range.

[Lucene has a related sparse\-indexing capability for sorted data](https://www.elastic.co/search-labs/blog/apache-lucene-10-release-highlights), but Elasticsearch does not expose it as a separate storage category in the [\_disk\_usage](https://www.elastic.co/docs/reference/elasticsearch/rest-apis/index-disk-usage) output used for this comparison.

Also note that in Elasticsearch the inverted index is not used only for full\-text search. For all string fields except `Body`, we used the [keyword](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/keyword) type, while `Timestamp` uses a [date](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/date) type and numeric fields use [numeric](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/number) types. All these types by default also populate the inverted index for exact\-match filtering, with values indexed as\-is and without analysis. This means filters on these fields can be resolved efficiently through the inverted index, making it the closest analogue to ClickHouse’s sparse primary index in this comparison.

### Points and norms for filtering and scoring [\#](/blog/elasticsearch-log-analytics-clickhouse#points-and-norms-for-filtering-and-scoring)

In Elasticsearch, **④ [points](https://www.elastic.co/fr/blog/lucene-points-6-0)** support efficient filtering and range queries on numeric and date fields, while [norms](https://www.elastic.co/docs/reference/elasticsearch/mapping-reference/norms) store per\-document scoring factors for text relevance.

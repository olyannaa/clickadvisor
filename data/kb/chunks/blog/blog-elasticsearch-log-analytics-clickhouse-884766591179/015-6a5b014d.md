At larger scales, Elasticsearch’s hot\-run advantage from its finer\-grained filter cache also appears to weaken. ## Bonus: Scaling the same workload in ClickHouse Cloud [\#](/blog/elasticsearch-log-analytics-clickhouse#bonus-scaling-the-same-workload-in-clickhouse-cloud)

In many analytical workloads, techniques such as [materialized views](https://clickhouse.com/docs/materialized-views) can pre\-aggregate data and reduce how much data must be scanned by queries.

Log analytics is different.

During incident response, **engineers often need the original raw events**: search for an error message, inspect surrounding logs, isolate affected hosts, break results down by service, and pivot repeatedly with new filters. That makes aggressive pre\-aggregation far less useful than in traditional BI workloads.

When raw data must still be queried, the practical way to go faster is simple: add more compute.

In [ClickHouse Cloud](https://clickhouse.com/cloud), compute is separated from storage, so additional replica nodes (replicating compute, not data) can join instantly without reshuffling data first. All nodes read from the same shared object storage.

[Parallel replicas](https://clickhouse.com/docs/deployment-guides/parallel-replicas) make query execution [scale horizontally](https://clickhouse.com/blog/clickhouse-parallel-replicas) by distributing data reads across the replica node fleet.

Loading video...As more nodes participate, the same query can fan out across more machines, reducing runtime. The chart below shows the effect on the 50B\-row query suite.

![Blog-FTS-vs-Elasticsearch.007.png](/uploads/Blog_FTS_vs_Elasticsearch_007_40ed1c9ddb.png)
Compared with the OSS single\-node baseline ([19\.1s](https://github.com/ClickHouse/TextBench/blob/main/clickhouse/results/parallel_replicas/50B.OSS.json)), the same workload drops to [12\.5s](https://github.com/ClickHouse/TextBench/blob/main/clickhouse/results/parallel_replicas/50B.aws.120.3.parallel_replicas.json) on 3 nodes, [6\.45s](https://github.com/ClickHouse/TextBench/blob/main/clickhouse/results/parallel_replicas/50B.aws.120.9.parallel_replicas.json) on 9 nodes, and [3\.27s](https://github.com/ClickHouse/TextBench/blob/main/clickhouse/results/parallel_replicas/50B.aws.120.20.parallel_replicas.json) on 20 nodes. The practical effect is that log search and analysis feel much more interactive, whether that is a human exploring [ClickStack](https://clickhouse.com/clickstack) dashboards or an agent querying logs in a loop.

This becomes even more powerful with [index sharding](https://clickhouse.com/blog/index-sharding-clickhouse-cloud-petabyte-scale-indexing). As the storage section [showed](/blog/elasticsearch-log-analytics-clickhouse#storage-breakdown-by-data-structure), the full\-text index on **the 50B OTel dataset is already about half a TiB (515\.78 GiB)**. Index sharding spreads the analysis of that index across the replica fleet, mirroring how parallel replicas already spread data reads.

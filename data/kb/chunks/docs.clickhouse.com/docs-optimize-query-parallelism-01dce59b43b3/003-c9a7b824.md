### Load balancing across processing lanes[​](#load-balancing-across-processing-lanes "Direct link to Load balancing across processing lanes")

Note that the `Resize` operators in the physical plan above [repartition and redistribute](/docs/academic_overview#4-2-multi-core-parallelization) data block streams across processing lanes to keep them evenly utilized. This rebalancing is especially important when data ranges vary in how many rows match the query predicates, otherwise, some lanes may become overloaded while others sit idle. By redistributing the work, faster lanes effectively help out slower ones, optimizing overall query runtime.

## Why max\_threads isn't always respected[​](#why-max-threads-isnt-always-respected "Direct link to Why max_threads isn't always respected")

As mentioned above, the number of `n` parallel processing lanes is controlled by the `max_threads` setting, which by default matches the number of CPU cores available to ClickHouse on the server:

```
SELECT getSetting('max_threads');

```

```
   ┌─getSetting('max_threads')─┐
1. │                        59 │
   └───────────────────────────┘

```

However, the `max_threads` value may be ignored depending on the amount of data selected for processing:

```
EXPLAIN PIPELINE
SELECT
   max(price)
FROM
   uk.uk_price_paid_simple
WHERE town = 'LONDON';

```

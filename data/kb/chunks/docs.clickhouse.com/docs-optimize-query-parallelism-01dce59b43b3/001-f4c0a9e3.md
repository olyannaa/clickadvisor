# How ClickHouse executes a query in parallel \| ClickHouse Docs

- - [Performance and optimizations](/docs/operations/overview)- Query parallelism
[Edit this page](https://github.com/ClickHouse/clickhouse-docs/blob/main/docs/guides/best-practices/query-parallelism.md)# How ClickHouse executes a query in parallel

ClickHouse is [built for speed](/docs/concepts/why-clickhouse-is-so-fast). It executes queries in a highly parallel fashion, using all available CPU cores, distributing data across processing lanes, and often pushing hardware close to its limits.

This guide walks through how query parallelism works in ClickHouse and how you can tune or monitor it to improve performance on large workloads.

We use an aggregation query on the [uk\_price\_paid\_simple](/docs/parts) dataset to illustrate key concepts.

## Step\-by\-step: How ClickHouse parallelizes an aggregation query[​](#step-by-step-how-clickHouse-parallelizes-an-aggregation-query "Direct link to Step-by-step: How ClickHouse parallelizes an aggregation query")

When ClickHouse ① runs an aggregation query with a filter on the table's primary key, it ② loads the primary index into memory to ③ identify which granules need to be processed, and which can be safely skipped:

![Index analysis](/docs/assets/images/query-parallelism_01-04167e71e6fb8ce3a9b12a2c11256d4a.gif)
### Distributing work across processing lanes[​](#distributing-work-across-processing-lanes "Direct link to Distributing work across processing lanes")

The selected data is then [dynamically](#load-balancing-across-processing-lanes) distributed across `n` parallel [processing lanes](/docs/academic_overview#4-2-multi-core-parallelization), which stream and process the data [block](/docs/development/architecture#block) by block into the final result:

![4 parallel processing lanes](/docs/assets/images/query-parallelism_02-82fba4a0fd87b980b12a61b59f4533f4.gif)
  

The number of `n` parallel processing lanes is controlled by the [`max_threads`](/docs/operations/settings/settings#max_threads) setting, which by default matches the number of cores (threads) of a single CPU available to ClickHouse on the server. In the example above, we assume `4` cores.

On a machine with `8` cores, query processing throughput would roughly double (but memory usage would also increase accordingly), as more lanes process data in parallel:

![8 parallel processing lanes](/docs/assets/images/query-parallelism_03-2120da7d58c366bd09295e0bc6bc9b20.gif)
  

Efficient lane distribution is key to maximizing CPU utilization and reducing total query time.

### Processing queries on sharded tables[​](#processing-queries-on-sharded-tables "Direct link to Processing queries on sharded tables")

When table data is distributed across multiple servers as [shards](/docs/shards), each server processes its shard in parallel. Within each server, the local data is handled using parallel processing lanes, just as described above:

![Distributed lanes](/docs/assets/images/query-parallelism_04-fdd68c179689c7410287062a72440de8.gif)
  

The server that initially receives the query collects all sub\-results from the shards and combines them into the final global result.

Distributing query load across shards allows horizontal scaling of parallelism, especially for high\-throughput environments.

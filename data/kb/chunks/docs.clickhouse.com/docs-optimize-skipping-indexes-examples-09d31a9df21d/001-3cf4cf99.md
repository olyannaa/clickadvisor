This page consolidates ClickHouse data skipping index examples, showing how to declare each type, when to use them, and how to verify they're applied. All features work with [MergeTree\-family tables](/docs/engines/table-engines/mergetree-family/mergetree).

**Index syntax:**

```
INDEX name expr TYPE type(...) [GRANULARITY N]

```

ClickHouse supports six skip index types:

| Index Type Description| **minmax** Tracks minimum and maximum values in each granule| **set(N)** Stores up to N distinct values per granule| **text** Inverted index over tokenized string data for full text search| **bloom\_filter(\[false\_positive\_rate])** Probabilistic filter for existence checks| **ngrambf\_v1** N\-gram bloom filter for substring searches| **tokenbf\_v1** Token\-based bloom filter for full\-text searches | | | | | | | | | | | | | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | | |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |


Each section provides examples with sample data and demonstrates how to verify index usage in query execution.

## MinMax index[​](#minmax-index "Direct link to MinMax index")

The`minmax` index is best for range predicates on loosely sorted data or columns correlated with `ORDER BY`.

```
-- Define in CREATE TABLE
CREATE TABLE events
(
  ts DateTime,
  user_id UInt64,
  value UInt32,
  INDEX ts_minmax ts TYPE minmax GRANULARITY 1
)
ENGINE=MergeTree
ORDER BY ts;

-- Or add later and materialize
ALTER TABLE events ADD INDEX ts_minmax ts TYPE minmax GRANULARITY 1;
ALTER TABLE events MATERIALIZE INDEX ts_minmax;

-- Query that benefits from the index
SELECT count() FROM events WHERE ts >= now() - 3600;

-- Verify usage
EXPLAIN indexes = 1
SELECT count() FROM events WHERE ts >= now() - 3600;

```

See a [worked example](/docs/best-practices/use-data-skipping-indices-where-appropriate#example) with `EXPLAIN` and pruning.

## Set index[​](#set-index "Direct link to Set index")

Use the `set` index when local (per\-block) cardinality is low; not helpful if each block has many distinct values.

The `ngrambf_v1` index splits strings into n\-grams. It works well for `LIKE '%...%'` queries. It supports String/FixedString/Map (via mapKeys/mapValues), as well as tunable size, hash count, and seed. See the documentation for [N\-gram bloom filter](/docs/engines/table-engines/mergetree-family/mergetree#n-gram-bloom-filter) for further details.

```
-- Create index for substring search
ALTER TABLE logs ADD INDEX msg_ngram msg TYPE ngrambf_v1(3, 10000, 3, 7) GRANULARITY 1;
ALTER TABLE logs MATERIALIZE INDEX msg_ngram;

-- Substring search
SELECT count() FROM logs WHERE msg LIKE '%timeout%';

EXPLAIN indexes = 1
SELECT count() FROM logs WHERE msg LIKE '%timeout%';

```

[This guide](/docs/use-cases/observability/schema-design#text-index-for-full-text-search) shows practical examples and when to use token vs ngram.

**Parameter optimization helpers:**

The four ngrambf\_v1 parameters (n\-gram size, bitmap size, hash functions, seed) significantly impact performance and memory usage. Use these functions to calculate optimal bitmap size and hash function count based on your expected n\-gram volume and desired false positive rate:

```
CREATE FUNCTION bfEstimateFunctions AS
(total_grams, bits) -> round((bits / total_grams) * log(2));

CREATE FUNCTION bfEstimateBmSize AS
(total_grams, p_false) -> ceil((total_grams * log(p_false)) / log(1 / pow(2, log(2))));

-- Example sizing for 4300 ngrams, p_false = 0.0001
SELECT bfEstimateBmSize(4300, 0.0001) / 8 AS size_bytes;  -- ~10304
SELECT bfEstimateFunctions(4300, bfEstimateBmSize(4300, 0.0001)) AS k; -- ~13

```

See [parameter docs](/docs/engines/table-engines/mergetree-family/mergetree#n-gram-bloom-filter) for complete tuning guidance.

## Token Bloom filter (tokenbf\_v1\) for word\-based search *(Deprecated)*[​](#token-bloom-filter-tokenbf-v1-for-word-based-search "Direct link to token-bloom-filter-tokenbf-v1-for-word-based-search")

NoteThe usage of `tokenbf_v1` indexes for full\-text search is deprecated in ClickHouse versions `>= 26.2` in favor of `text` indexes (see [here](/docs/engines/table-engines/mergetree-family/textindexes) for further details).

`tokenbf_v1` indexes tokens separated by non\-alphanumeric characters. You should use it with [`hasToken`](/docs/sql-reference/functions/string-search-functions#hasToken), `LIKE` word patterns or equals/IN. It supports `String`/`FixedString`/`Map` types.

See [Token bloom filter](/docs/engines/table-engines/mergetree-family/mergetree#token-bloom-filter) and [Bloom filter types](/docs/optimize/skipping-indexes#skip-index-types) pages for more details.

```
ALTER TABLE logs ADD INDEX msg_token lower(msg) TYPE tokenbf_v1(10000, 7, 7) GRANULARITY 1;
ALTER TABLE logs MATERIALIZE INDEX msg_token;

-- Word search (case-insensitive via lower)
SELECT count() FROM logs WHERE hasToken(lower(msg), 'exception');

EXPLAIN indexes = 1
SELECT count() FROM logs WHERE hasToken(lower(msg), 'exception');

```

See observability examples and guidance on token vs ngram [here](/docs/use-cases/observability/schema-design#text-index-for-full-text-search).
